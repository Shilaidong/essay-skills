# 图表、流程图、甘特图与图片

## 1. 先问：图表解决什么问题

| 读者任务 | 推荐视觉 |
|---|---|
| 比较方案 | 对比表、雷达图（谨慎）、二维矩阵 |
| 看趋势 | 折线图 |
| 看构成 | 条形图、堆叠条形图；饼图仅少量类别 |
| 看排名 | 排序条形图 |
| 看关系 | 散点图、关系图 |
| 看流程 | 流程图 |
| 看时间 | 时间轴、甘特图 |
| 看责任 | RACI 表 |
| 看风险 | 概率×影响矩阵 |
| 看地理 | 地图，需真实空间数据 |

## 2. 数据图表标准

- 数据来源明确；
- 单位明确；
- 坐标轴不误导；
- 不截断坐标轴造成夸大，除非有充分说明；
- 类别排序有逻辑；
- 避免 3D 图；
- 避免装饰网格和过多标签；
- 对不确定性使用区间或注释；
- 真实数据图必须可复现。

## 3. 甘特图数据格式

`timeline.csv` 推荐列：

```text
section,task,start,end,status,milestone,owner,notes
```

- `start/end`：YYYY-MM-DD；
- `milestone`：true/false；
- `status`：planned/in_progress/done/blocked；
- 里程碑可令 start=end；
- 任务过多时拆成阶段，不要把每个微任务都放进图。

## 4. 甘特图生成

```bash
python scripts/render_gantt.py timeline.csv \
  --output gantt.png \
  --title "项目实施时间轴" \
  --subtitle "里程碑与关键工作流"
```

在 Word 中：

- 插入高清 PNG；
- 横向内容可单独横向分节；
- 图下保留 1–2 句阅读说明；
- 不插入 Mermaid 源码；
- 若用户需要源码，作为单独附件交付。

## 5. 流程图

流程图应：

- 起点和终点明确；
- 每个节点动词化；
- 分支条件写在连线上；
- 不超过 7–9 个主节点；
- 复杂流程拆成主流程和子流程。

可使用代码生成 SVG/PNG，也可在 Word 中用形状。最终必须渲染检查。

## 6. 决策矩阵

推荐结构：

| 方案 | 适用条件 | 优势 | 代价 | 风险 | 选择触发条件 |
|---|---|---|---|---|---|

若需要评分：

- 先解释标准；
- 分数不能替代判断；
- 权重来自用户优先级；
- 对高度主观标准给出注释。

## 7. 图片

### 用户图片

- 不改变关键事实；
- 裁剪前保留原始文件；
- 检查人像和隐私；
- 不未经同意公开未成年人；
- 图注说明时间、地点或来源。

### 在线图片

- 优先官方媒体库、公共领域或明确授权素材；
- 记录来源和许可；
- 不直接抓取模糊缩略图；
- 不让图片成为唯一证据。

### 生成式图片

适合：

- 封面概念图；
- 抽象场景；
- 非事实性插图。

不适合：

- 模拟真实数据；
- 伪造现场照片；
- 伪造人物、证书、产品或研究结果；
- 替代真实设计图或工程图。

## 8. 图表 QA

检查：

- 标题是否与数据一致；
- 图例是否完整；
- 字号是否可读；
- 颜色在黑白打印时是否可区分；
- 数据点是否与源数据一致；
- 图注是否包含来源；
- Word 中是否被压缩或拉伸；
- 不同页面是否使用一致的视觉语法。
