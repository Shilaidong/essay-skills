# 工具路由、调用顺序与失败处理

## 1. 能力优先，而不是工具名称优先

不同宿主对工具命名不同。本 Skill 以能力描述工具：

- 搜索网页；
- 打开网页；
- 读取文件；
- 写入文件；
- 执行 Shell；
- 执行 Python；
- 访问私有连接器/MCP；
- 创建或编辑 DOCX；
- 渲染 DOCX；
- 生成数据图表；
- 生成装饰性图片。

在 Codex 或 OpenCode 中，先查看当前会话已授权的工具，再选择实现方式。不要把某个平台的私有工具名写死为唯一方案。

## 2. 推荐调用顺序

### 2.1 新建研究型 Word 报告

```text
1. read / glob / shell：盘点输入材料
2. PDF/表格解析：提取结构化信息
3. web search/fetch：补充并核验当前事实
4. source ledger：记录来源与状态
5. Python：清洗数据、生成表格或图表
6. 写 report.yaml
7. build_docx.py：生成 DOCX
8. qa_docx.py：渲染为逐页 PNG
9. image viewer：逐页检查
10. 修改并重新生成
11. 最终交付 DOCX
```

### 2.2 修改已有 Word

```text
1. 复制原文件到工作副本
2. 解析文档文本、样式和图片
3. 明确修改清单
4. python-docx 或专门 DOCX 工具编辑
5. 需要时做 OOXML 定向修补
6. 渲染并逐页对比
7. 清除评论、修订和元数据（按用户要求）
8. 交付新文件，不覆盖原件
```

### 2.3 从 PDF 或网页生成报告

```text
1. 优先提取原生文本
2. 对图表、扫描页和复杂表格做页面渲染
3. 只在必要时 OCR
4. 将每个事实绑定到页码或 URL
5. 再进入分析与 DOCX 工作流
```

## 3. 并行化

适合并行：

- 互不依赖的多个官方来源检索；
- 不同输入文件的盘点；
- 多个章节的事实核验；
- 图表与文字的独立准备。

不适合并行：

- 第二步依赖第一步确认的实体名称；
- 多人同时修改同一个 DOCX；
- 在数据口径未统一前分别计算同一指标；
- 在目录未定前大量撰写正文。

并行结果汇总时统一：

- 时间口径；
- 术语；
- 文件版本；
- 来源格式；
- 数字单位。

## 4. Shell 调用规范

- 使用非交互命令；
- 显式指定工作目录；
- 对包含空格的路径加引号；
- 先 `ls` 或 `find` 确认文件；
- 危险命令前检查目标路径；
- 不使用宽泛的 `rm -rf` 指向用户目录；
- 将输出写入新目录；
- 长命令拆成可复核步骤；
- 记录 exit code 和 stderr。

## 5. Python 调用规范

- 脚本应可从命令行运行；
- 参数使用 `argparse`；
- 对输入文件、列名和数据类型做验证；
- 失败时给出可操作错误信息；
- 固定随机种子（若涉及抽样）；
- 图表标注数据来源；
- 输出前创建目录；
- 不静默覆盖文件，除非有 `--force`；
- 关键步骤写日志。

## 6. Web 工具调用规范

- 查询包含实体和具体事实，不使用过长问题句；
- 一次搜索少量高相关查询；
- 优先 `site:` 限定官方域名；
- 打开正文页面核验；
- 使用查找功能定位关键词；
- 对 PDF 使用截图查看图表；
- 若页面被脚本阻挡，寻找同一机构的 PDF、公告或存档页；
- 不引用搜索摘要作为唯一依据。

## 7. 私有连接器 / MCP

- 用户提供私有链接时优先使用匹配连接器；
- 不把私有链接交给公开搜索引擎；
- 只读取完成任务所需范围；
- 写操作前确认用户意图；
- 返回的文件引用按工具契约传递，不转成猜测路径；
- 涉及邮件、日历或云盘时，默认只读，除非用户明确要求写入。

## 8. DOCX 工具策略

### 8.1 生成

优先 `python-docx` 或本包 `build_docx.py`，因为可复现、可配置、可再次生成。

### 8.2 复杂功能

以下情况可能需要 OOXML：

- 真正的目录字段；
- 交叉引用；
- 批注；
- 修订；
- 内容控件；
- 复杂页眉页脚；
- 某些表格属性。

使用 OOXML 后必须再次渲染并结构检查。

### 8.3 渲染

优先使用宿主提供的标准 DOCX 渲染器。若没有：

1. LibreOffice 无头转换为 PDF；
2. Poppler 转成 PNG；
3. 逐页检查；
4. 生成 contact sheet 便于总览。

## 9. 图像工具策略

- 真实数据图表：Python；
- 流程、时间轴、矩阵：可复现绘图或形状；
- 用户提供图片：尽量保留原始比例和分辨率；
- 装饰图：仅在用户明确需要时使用生成式图片；
- 不用生成式图片模拟真实人物、真实场地、数据或证据；
- 图片进入 Word 前检查版权、分辨率、色彩和说明文字。

## 10. 失败处理

### 搜索不到

记录：

- 已尝试关键词；
- 已查来源；
- 找到的最接近证据；
- 为什么不足；
- 文档中如何降级表述。

### 文件损坏

- 保留原件；
- 尝试解压检查；
- 使用替代解析器；
- 请求用户重新导出；
- 不伪造无法读取的内容。

### 字体缺失

- Word 样式仍指定目标字体；
- 本地渲染可使用可用 CJK 替代字体；
- 不分发字体文件；
- 在 QA 备注中说明本机替代渲染。

### 渲染失败

- 使用独立可写 HOME 和 LibreOffice profile；
- 检查文件是否被占用；
- 检查输出 PDF 是否生成；
- 必要时重新保存 DOCX；
- 不绕过视觉 QA。

## 11. 交付声明

只声明实际完成的步骤。例如：

- “已核验官方来源并写入来源表”；
- “已生成并逐页渲染检查 DOCX”；
- “甘特图已作为 PNG 插入 Word”。

不要写“全部验证无误”，除非确实逐项检查。
